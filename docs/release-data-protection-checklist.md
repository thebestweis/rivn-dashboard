# RIVN OS: чеклист бэкапов и защиты перед релизом

## 1. Бэкапы Supabase

Минимум перед релизом:

- Включить автоматические бэкапы проекта Supabase.
- Перед крупными изменениями запускать ручной бэкап или SQL dump.
- Хранить отдельную копию схемы и критичных таблиц: `workspaces`, `workspace_members`, `profiles`, `clients`, `projects`, `tasks`, `payments`, `expenses`, `payroll_payouts`, `workspace_billing`, `billing_transactions`, `referral_*`, `crm_*`, `avito_report_*`.

## 2. Инструкция восстановления

Если что-то пошло не так:

1. Остановить деплой новых изменений.
2. Проверить, проблема в коде или данных.
3. Если проблема в коде, откатить приложение на предыдущий коммит и перезапустить сервер.
4. Если повреждены данные, восстановить нужные таблицы из последнего бэкапа.
5. После восстановления проверить вход, dashboard, CRM, Avito-отчёты, платежи и биллинг.

## 3. Service role

`SUPABASE_SERVICE_ROLE_KEY` нельзя хранить в публичном коде и нельзя отдавать на клиент.

Правила:

- Ключ должен быть только в `.env.production` на сервере и в закрытых env-переменных.
- Не логировать service role key.
- Не использовать service role в клиентских компонентах.
- Все API с service role должны явно проверять `workspaceId` или права super admin.

## 4. RLS

Перед релизом проверить:

- RLS включён на пользовательских таблицах.
- Пользователь видит только свои кабинеты и данные своего workspace.
- Менеджер по продажам видит только свои сделки, если это заложено ролью.
- Service role endpoints дополнительно фильтруют данные по `workspace_id`.

## 5. Критичные действия

Логировать в `admin_action_logs`:

- ручное изменение баланса;
- активацию и продление тарифа;
- ручную смену статуса тарифа;
- ручную реферальную привязку;
- изменение статуса реферальной выплаты;
- массовую рассылку уведомлений;
- будущие критичные действия: удаление данных, массовый импорт, ручной запуск отчётов.

## 6. Health-check перед релизом

В админке проверить блок `Health-check сервиса`:

- Supabase доступен;
- Telegram отвечает;
- Avito token получается;
- ключевые env-переменные заполнены;
- есть свежие логи cron/отчётов.

Если один из пунктов красный или жёлтый, сначала разобраться с ним, потом запускать релиз.
